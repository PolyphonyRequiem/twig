## Executive Summary

This plan addresses four improvement findings from the Help System (Epic 1522) close-out review, tracked as Issue #1668 under the Closeout Findings epic (#1603). The four child tasks cover: (1) automated validation of generated command examples against actual CLI output, (2) adding worktree detection to the close-out prompt so it gracefully handles linked-worktree environments, (3) adding diff summary instructions to the close-out prompt for mechanical changes spanning many commands, and (4) adding a summary-only close-out appendix to plan documents to reduce their size. Task 1 creates a new PowerShell validation script; Tasks 2, 3, and 4 modify the close-out agent's prompt. No C# code changes are required. The expected outcome is a more reliable close-out process with automated guardrails for example accuracy, worktree-aware operations, concise diff reporting, and leaner plan documents.

### Context for These Findings

### Help Output Format (Validation Parsing Context)

The validation script (Task 1) must parse two distinct output formats produced by the twig CLI:

**Top-level help (`twig --help`) — GroupedHelp format** (`Program.cs`, lines 993–1086):
```
twig <version>

Usage: twig [command] [-h|--help] [--version]

<Category>:
  <command>             <description>.
  <compound cmd>        <description>.
```
Category headers are bare words ending with `:` (e.g., `Getting Started:`, `Views:`, `Navigation:`). Command entries are indented 2 spaces, followed by 2+ spaces of padding before the description. Compound commands (e.g., `nav up`, `seed new`) appear as space-separated tokens in the same indented line. Many entries include positional-argument placeholders (e.g., `<title>`, `[pattern]`, `<s> <t>`) or flag variants (e.g., `--all`, `--editor`, `--batch`) as additional tokens after the command name — approximately 15 of 64 entries (23.4%) have 3+ tokens before the padding. The format is hand-maintained in `GroupedHelp.Show()`, not generated by ConsoleAppFramework.

**Per-command help (`twig <cmd> --help`) — ConsoleAppFramework format**:
```
Usage: twig <command> [arguments] [options]

<description>

Options:
  --flag-name <type>  Description
  -h, --help          Show help message and exit

Examples:
  twig <command> --flag value   Description
```
The `Options:` section is generated by ConsoleAppFramework. The `Examples:` section is appended by `CommandExamples.ShowIfPresent()` (line 134 of `Program.cs`), which prints a blank line, an `Examples:` header, and each example indented 2 spaces.

## Problem Statement

Four distinct gaps exist in the SDLC workflow tooling:

1. **No automated validation of command examples**: `CommandExamples.cs` contains 60 hardcoded example entries (e.g., `"twig set 1234    Set active context to work item #1234"`). These examples are validated by `GroupedHelpTests.AllCommands_HaveExamples()` for existence and count (≥2 per command), but there is **no validation that the example syntax matches actual CLI output**. If a command's flags or arguments change, the examples can silently drift out of sync. The architect/coder agents generate these examples during implementation, but no automated gate catches drift during subsequent modifications.

2. **No worktree detection in the close-out prompt**: The close-out agent's Step 1 instructs `git checkout main && git pull` to verify merged commits. In a linked git worktree, `git checkout main` fails if `main` is already checked out in the primary worktree (git prohibits the same branch being checked out in multiple worktrees simultaneously). The prompt has no convention for detecting this scenario. Note: `git branch --no-merged main` (Step 1b) works correctly from any worktree — branch listing is repository-wide, not per-worktree. The issue is specifically with checkout operations. The `FlowCloseCommand` already handles worktrees by detecting linked worktrees and skipping branch cleanup (lines 176–186 of `FlowCloseCommand.cs`), but the close-out agent prompt does not apply the same pattern.

3. **No diff summaries for mechanical changes**: When a coder agent makes wide, mechanical changes across many commands (e.g., adding `--output json` to every command, renaming a parameter across 50 files), the close-out agent's Step 6 (meta-observations) has no concise way to summarize the scope. The git log shows individual commits, but there is no aggregate summary like "Modified 47 files, 94 lines changed across 23 commands." The observations become verbose without adding clarity. **Note**: The diff range must use the previous release tag as a baseline (e.g., `git diff --stat <prev_tag>..HEAD`), not `origin/main..HEAD`, because after Step 1's `git checkout main && git pull`, HEAD equals `origin/main` and the range would be empty. Step 10 creates a new tag at close-out completion, so the latest existing tag during Steps 5–6 always represents the pre-epic baseline.

4. **Plan documents grow unbounded**: Close-out output (Steps 5–6) is inlined into the plan document as a completion section (Step 4). For large epics with many PR groups, the observations text can add 2–5 KB to an already-large plan. Plan documents like `companion-auto-upgrade.plan.md` (72 KB) are already at the upper end of usability. A summary-only appendix pattern would keep the plan lean while preserving the full observations in ADO notes (Step 7).

## Goals and Non-Goals

### Goals

1. **G1 — Example validation**: Create an automated validation script that runs `twig <command> --help` for each command with registered examples and verifies the example flag syntax is consistent with the actual CLI output
2. **G2 — Worktree-aware close-out**: Add worktree detection to the close-out prompt so that checkout and branch operations degrade gracefully when running in a linked worktree
3. **G3 — Diff summaries**: Add prompt instructions for the close-out agent to produce aggregate diff statistics (files changed, lines added/removed, commits) for mechanical change sets
4. **G4 — Summary appendix**: Restructure the close-out agent's plan-update step (Step 4) to write a concise completion summary (≤10 lines) in the plan body and emit full observations only to the ADO note (Step 7)

### Non-Goals

- **NG1**: Modifying `CommandExamples.cs` content — this plan validates examples, not rewrites them
- **NG2**: Adding worktree support to the C# CLI beyond what already exists — `FlowCloseCommand` and `GitCliService` already handle worktrees
- **NG3**: Changing the closeout-filing workflow (`closeout-filing.yaml`) — filing is downstream and consumes the same output fields regardless
- **NG4**: Changing the YAML workflow schema (`twig-sdlc.yaml` agent definitions, routing, or output fields)
- **NG5**: Retroactively updating existing plan documents with the new appendix format

### Key Components

#### Component 1: `Validate-CommandExamples.ps1`

A PowerShell script (~150 lines) that:
1. Builds the twig binary (`dotnet build src/Twig`); if the build fails, exit immediately with a clear error message — do not proceed to help parsing
2. Discovers all commands by running `twig --help` and parsing the GroupedHelp output
3. For each command, runs `twig <command> --help` and captures output
4. Extracts `--flag` tokens from the Options section and from each example line
5. Reports mismatches and exits 1 if any example flag doesn't appear in the Options section

**Parsing strategy** (addresses [Background: Help Output Format](#help-output-format-validation-parsing-context)):

1. **Command discovery** from `twig --help`: Iterate lines, detect category headers (line matching `^\w[\w\s]*:$`). Under each category, extract command entries from indented lines matching `^\s{2}(\S+(?:\s+\S+)*)\s{2,}` — the first capture group contains all tokens before the description padding (the command name plus any positional placeholders or flag variants). After extraction, strip non-command tokens:
   - Remove placeholder tokens matching `<[^>]+>` (e.g., `<title>`, `<id|pattern>`, `<s>`)
   - Remove optional-argument tokens matching `\[[^\]]+\]` (e.g., `[pattern]`, `[id]`, `[val]`)
   - Remove flag-variant tokens matching `--[a-z][a-z0-9-]*` (e.g., `--all`, `--editor`, `--batch`)
   - Trim and collapse whitespace to produce the bare command name (e.g., `seed new`, `config`, `link parent`)
   - Deduplicate: entries like `seed new <title>` and `seed new --editor` both resolve to `seed new` — run validation once per unique command.

2. **Flag extraction** from `twig <cmd> --help`: Locate the `Options:` section header. For each subsequent indented line until the next section header or end-of-output, extract long-form flags matching `(--[a-z][a-z0-9-]*)` — this captures tokens like `--output`, `--force`, `--no-refresh`. Stop collecting when hitting `Examples:` or a blank-line-followed-by-non-indented-text. If no `Options:` section is present (some commands may have only usage and examples), treat the valid flag set as empty — any `--flag` in the examples will be reported as a mismatch.

3. **Example flag extraction**: Locate the `Examples:` section header (appended by `CommandExamples.ShowIfPresent()`). For each subsequent indented line, extract `--flag` tokens using the same `(--[a-z][a-z0-9-]*)` pattern.

4. **Comparison**: For each command, every `--flag` found in Examples must also appear in Options. Report mismatches as: `MISMATCH: <command> example uses '--flag' but Options section does not list it`.

**Scope**: Long-form `--flag` tokens only. Positional args, short flags (e.g., `-o`), and bare values are out of scope for the initial version. Command/example discovery is done entirely from the compiled binary's `--help` output — not from parsing `CommandExamples.cs` source.

#### Component 2: Close-Out Prompt — Worktree-Aware Operations (Step 1)

Replace the unconditional `git checkout main && git pull` with a `rev-parse` worktree check:
- If `--git-common-dir` ≠ `--git-dir` (linked worktree): use `git fetch origin main` + verify via `git log origin/main`
- Otherwise: keep existing `git checkout main && git pull` behavior

Step 1b (`git branch --no-merged main`) is unaffected — it works from any worktree.

#### Component 3: Close-Out Prompt — Aggregate Diff Statistics (Steps 5–6)

Add a diff statistics block to Step 5 that uses the previous release tag as the baseline:

```bash
# Get the most recent release tag (created by the previous close-out's Step 10)
baseline=$(git tag -l "v*" --sort=-version:refname | head -1)
# Capture aggregate diff stats from baseline to current HEAD
git diff --stat "$baseline"..HEAD
# Capture commit count
git rev-list --count "$baseline"..HEAD
```

This approach is correct because Step 10 creates a new version tag *after* Steps 5–6 complete, so during observation generation the latest existing tag represents the pre-epic baseline. The agent formats results into a summary table (files changed, lines ±, commits) and references it in Step 6 for mechanical change descriptions.

**Fallback**: If no version tag exists (new repository) or `git diff` fails (shallow clone, detached HEAD), the prompt instructs the agent to skip the statistics table and note "diff stats unavailable — no baseline tag found" in observations.

#### Component 4: Close-Out Prompt — Summary-Only Plan Completion (Step 4)

Constrain the plan completion block to ≤10 lines (date, 1–2 sentence summary, ADO reference). Explicitly prohibit inlining full observations into the plan — that content belongs in the ADO note (Step 7). Update the "Mark all Epics and Tasks as DONE" instruction to use context-appropriate terminology: the current text assumes Epic-level close-out with plan-document "Epics" (which map to ADO Issues). Add a parenthetical noting that the agent should use the terminology matching the item type being closed (e.g., "Issues and Tasks" for Issue-level, "Epics and Tasks" for Epic-level with plan-document epic sections).

### Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | **PowerShell script vs. C# xUnit test** for example validation | The validation invokes the compiled AOT binary (behavior may differ from in-process). It matches the existing tooling pattern (`seed-from-plan.ps1`), runs standalone or in CI without test framework overhead, and complements `GroupedHelpTests.AllCommands_HaveExamples()` which validates example *existence* — this script validates example *accuracy*. |
| D2 | **Prompt edit vs. workflow YAML** for Tasks 2, 3, and 4 | The YAML workflow controls agent sequencing, routing, and output schema; the prompt controls agent *behavior within* a step. All three changes are behavioral nuances (which git commands to run, how to produce observations, how to format the completion block) — none require new output fields or routing logic. |
| D3 | **Long-form flags only** for validation scope | Validating only `--long-flag` tokens (not short flags like `-o` or positional args) reduces false-positive risk: short flags are aliases defined in ConsoleAppFramework and aren't always shown in examples, while positional args vary by usage pattern. Long-form flags have a 1:1 mapping between Options and Examples sections, making mismatches unambiguous. This can be extended later if the initial scope proves insufficient. |
| D4 | **`rev-parse` comparison vs. `git worktree list`** for worktree detection | `git rev-parse --git-common-dir` vs `--git-dir` is the same detection mechanism used by `GitCliService.GetWorktreeRootAsync()` (line 218), providing consistency between the CLI and the close-out prompt. `git worktree list` is heavier (lists all worktrees, requires parsing) and may not be available in minimal git installations. The rev-parse approach is a single comparison returning a boolean signal. |
| D5 | **Tag-based range vs. `origin/main..HEAD`** for diff statistics | After Step 1's `git checkout main && git pull`, HEAD equals `origin/main`, making `origin/main..HEAD` an empty range. The previous release tag (from Step 10 of the prior close-out) provides the correct baseline. Alternative approaches (capturing stats pre-checkout, merge-base references) were considered but add ordering complexity or require knowing the feature branch names. |

## Risks and Mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|------|-----------|--------|------------|
| R1 | **Validation script false positives**: Example text may contain `--` as a separator or description text rather than a flag prefix, causing spurious mismatch reports | Medium | Low | Parse `--` tokens only when matching `--[a-z][a-z0-9-]*` (flag naming convention). |
| R2 | **ConsoleAppFramework help format changes**: If ConsoleAppFramework changes its `--help` output layout (Options section indentation, section headers), the validation script's regex-based parser may break silently | Low | Medium | Pin the ConsoleAppFramework version in `Directory.Packages.props`. The script's first action is a canary check: parse `twig status --help` and assert the `Options:` header exists — fail fast if the format is unrecognizable. |
| R3 | **Diff statistics unavailable**: If the workflow operates on a repository with no prior version tags (first-ever close-out) or a shallow clone, the tag-based range `git diff --stat <tag>..HEAD` will fail | Low | Low | The prompt instruction includes a fallback: "If no version tag exists or `git diff` fails, skip the statistics table and note 'diff stats unavailable — no baseline tag found' in observations." |
| R4 | **Build failure blocks validation**: If `dotnet build src/Twig` fails (broken code, missing SDK), the validation script could proceed with a stale or missing binary and produce misleading results | Low | Medium | The script checks the build exit code immediately after `dotnet build` and terminates with a clear error message (e.g., "Build failed — cannot validate examples against a stale binary") before attempting any `--help` parsing. |
