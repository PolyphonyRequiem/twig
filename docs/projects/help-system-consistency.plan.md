# Help System Consistency & Coverage

> **Status**: 🔨 In Progress | **Epic**: #1522 | **Revision**: R7 — Revised per tech/readability review (84/91)

## Executive Summary

This plan addresses three inconsistencies in the twig CLI help system: missing commands in `twig help`, format divergence between `twig help` and `twig <invalid>`, and unhelpful per-command help that shows `[0] <string>` instead of meaningful descriptions. A fourth issue — `save` listed in help despite being `[Hidden]` — is also corrected. The fix adds command routing to unify help paths, enriches all 58 non-hidden methods (57 in `TwigCommands` after hiding `Seed`, plus 1 in `OhMyPoshCommands`) with XML doc comments, and appends usage examples via a new `CommandExamples` class. Tests prevent regressions.

## Background

### Current State

> **Point-in-time reference**: Line numbers cited in this document refer to `Program.cs` as of commit `HEAD` at plan authoring time and will shift as edits are made. All references are paired with the method or class name they refer to — use the name as the stable anchor.

The twig CLI uses a three-layer help architecture:

| Layer | Responsibility | Location |
|-------|---------------|----------|
| `GroupedHelp.Show()` | Categorized help for `twig help`, `twig --help`, `twig` (no workspace) | `Program.cs`, `GroupedHelp` class (~line 670) |
| ConsoleAppFramework default | Flat alphabetical command list for unknown commands | Source-generated by ConsoleAppFramework v5.7.13 |
| ConsoleAppFramework per-command | `--help` for individual commands (`twig set --help`) | Source-generated from XML doc comments |

**`GroupedHelp.Show()`** is a hand-crafted string literal in `Program.cs` that organizes ~50 commands into 9 categories (Getting Started, Views, Context, Navigation, Work Items, Git, Workflow, System, Experimental). It is invoked when:
- `args` is `["-h"]`, `["--help"]`, or `["help"]` (early-exit block, ~line 98; requires `args.Length == 1`)
- `args` is empty and no `.twig/` workspace exists (smart-landing block, ~line 103)

**ConsoleAppFramework default help** is generated by the source generator and produces a flat alphabetical list. It fires when `app.Run(args)` encounters an unrecognized command. This path includes ALL registered commands (even those missing from `GroupedHelp`) and uses a different visual format.

**Per-command help** is generated from XML doc comments on `TwigCommands` methods. Most methods have only `<summary>` tags, so positional arguments show as `[0] <string>` with no description. The `Update()` method has three `<param>` tags (`format`, `file`, `stdin`) on its named options, but is missing tags for its two positional `[Argument]` parameters (`field`, `value`) and the common `output` parameter. This demonstrates that ConsoleAppFramework's source generator picks up `<param>` tags for named options, but does not yet confirm the display improvement for positional `[Argument]` parameters. ConsoleAppFramework derives the displayed argument name from the C# parameter name (e.g., `field` from `[Argument] string field`), so `<param>` tags add description text alongside the already-derived name — the `[0] <string>` display occurs because the source generator omits description text when no `<param>` tag is present, not because it cannot resolve the parameter name.

`GroupedHelp.Show()` is called from exactly two locations in `Program.cs` — the early-exit block (when `args` is `-h`/`--help`/`help`) and the smart-landing block (when `args` is empty and no `.twig/` workspace exists). Both call sites will benefit from refactoring with zero behavioral change.

### ConsoleAppFramework CLI Name Mapping

ConsoleAppFramework derives CLI command names from C# method names using two rules:

1. **Explicit `[Command("name")]` attribute**: The CLI name is the attribute value. Used for compound commands (e.g., `NavUp` → `[Command("nav up")]` → `nav up`) and hyphenated names (e.g., `FlowStart` → `[Command("flow-start")]` → `flow-start`).
2. **No `[Command]` attribute**: The CLI name is the method name lowercased. In this codebase, all such methods are single-word PascalCase (e.g., `Set` → `set`, `Sprint` → `sprint`, `Init` → `init`).

This distinction is critical for `KnownCommands` correctness (T-1523-3) and the completeness test (T-1523-4) — CLI names must be derived from `[Command]` attributes when present, not from the `GroupedHelp` display text or method name alone.

### ConsoleAppFramework Per-Command Help Model

ConsoleAppFramework v5 generates help from XML doc comments at compile time:
- `<summary>` → command description (already present on all methods)
- `<param name="x">Description.</param>` → argument description
- `<param name="x">-a, Description.</param>` → alias + description
- `[Argument]` parameters become positional; name is derived from C# param name
- No native support for usage examples in help footer
- When `--help` is passed, the generated `TryShowHelpOrVersion()` prints help and returns `true`, causing the generated `Run` method to `return` normally — it does **not** call `Environment.Exit()`. This means post-`app.Run()` code executes reliably after help display.

## Problem Statement

1. **Missing commands**: `query` and `new` are fully functional commands that users cannot discover via `twig help`. The `seed` bare command should be marked `[Hidden]` since `seed new` already exists in help.

2. **`save` listed despite `[Hidden]`**: The `save` command appears in the GroupedHelp Work Items section despite `Save()` being marked `[Hidden]`. This is a pre-existing inconsistency that confuses users about deprecated commands.

3. **Format inconsistency**: Running `twig help` shows a professionally grouped categorized layout. Running `twig foobar` shows ConsoleAppFramework's flat alphabetical list with different formatting. Users encountering typos see a degraded help experience.

4. **Unhelpful per-command help**: `twig set --help` shows `[0] <string>` because most `TwigCommands` methods lack `<param>` XML doc comments (the `Update` method is the sole exception with three `<param>` tags). Users cannot discover what arguments mean, what values are accepted, or see usage examples.

## Proposed Design

### Architecture Overview

The design introduces three coordinated changes to the help system:

**Routing flow:**
```
Program.cs (arg routing)
  ├─ args = ["help"] / ["-h"] / ["--help"]  →  GroupedHelp.Show()        [existing]
  ├─ args = ["<unknown>", ...]              →  GroupedHelp.ShowUnknown()  [NEW]
  ├─ args = ["<command>", ..., "--help"]    →  app.Run()                 [existing]
  │                                            then CommandExamples.Show [NEW, post-run]
  └─ args = ["<known-command>", ...]        →  app.Run()                 [existing]
```

> **Note on sequencing**: `app.Run()` and `CommandExamples.ShowIfPresent()` execute sequentially — first `app.Run()` prints ConsoleAppFramework's help, then post-run code appends examples. They do NOT execute concurrently.

**Class structure:**
```
GroupedHelp (refactored)
  ├─ KnownCommands: HashSet<string>        ← manually-maintained, validated by test
  ├─ Show(): void                          ← categorized help (save removed, query/new added)
  └─ ShowUnknown(cmd): void                ← error prefix + Show()

CommandExamples (new file: src/Twig/CommandExamples.cs)
  ├─ Examples: Dictionary<string, string[]> ← command → example strings
  └─ ShowIfPresent(args): void             ← resolves command name, appends examples if found
```

### Key Components

#### 1. `GroupedHelp` Refactoring (`Program.cs`)

The existing `GroupedHelp` static class is refactored to:
- **Add missing commands**: `query` under "Context" and `new` under "Work Items"
- **Remove `save`**: Listed in GroupedHelp despite `Save()` being marked `[Hidden]`. Remove it. `sync` is the replacement and is already listed twice (Getting Started and Work Items).
- **Expose `KnownCommands`**: A `static HashSet<string>` containing all top-level command names and sub-command prefixes (e.g., `"set"`, `"nav"`, `"nav up"`, `"seed new"`, `"ohmyposh"`, `"ohmyposh init"`, `"help"`, etc.). This is a **manually-maintained list** — not derived by parsing the help text string, and not using reflection. The rationale: AOT forbids reflection in production, and string-parsing GroupedHelp text is fragile. The maintenance burden is mitigated by the completeness test (T-1523-4) which will fail the build if a new command is added without updating KnownCommands.
- **Include `"help"` in `KnownCommands`**: The early-exit block only catches `args.Length == 1 && args[0] is "help"`. Multi-arg invocations like `twig help extra` bypass that check. Without `"help"` in `KnownCommands`, `IsKnownCommand` would return false and trigger `ShowUnknown("help")` — a confusing error message. Adding `"help"` to `KnownCommands` lets it fall through to `app.Run()` which handles it gracefully.
- **Add `ShowUnknown(string command)`**: Writes `Unknown command: '<command>'` to stderr, then calls `Show()` to stdout.

#### 2. Unknown Command Interception (`Program.cs`)

Before `app.Run(args)`, add routing logic:

```csharp
// Check if the first arg is a known command (not a flag)
if (args.Length > 0 && !args[0].StartsWith('-') && !IsKnownCommand(args))
{
    GroupedHelp.ShowUnknown(args[0]);
    Environment.ExitCode = 1;
    return;
}
```

The `IsKnownCommand` check examines `args[0]` (and `args[0..1]` for compound sub-commands like `nav up`) against `GroupedHelp.KnownCommands`. Since ConsoleAppFramework v5 expects the command name as the first argument(s), checking `args[0]` is correct — there is no need to handle flags before the command name. Edge cases:
- `twig nav up` → known (matches "nav up")
- `twig nav` → known (matches "nav")
- `twig foobar` → unknown → grouped help
- `twig --output json` → `args[0]` starts with `-`, skip interception; ConsoleAppFramework handles it

#### 3. Per-Command Help Enhancement (`Program.cs` — TwigCommands)

Add `<param>` XML doc comments to every parameter in every `TwigCommands` method, plus `OhMyPoshCommands.Init()`. The `Update()` method has three existing `<param>` tags (`format`, `file`, `stdin`) — this pattern is extended to all other non-hidden methods. After hiding `Seed`, there are 57 non-hidden `TwigCommands` methods plus 1 `OhMyPoshCommands` method (58 total). ConsoleAppFramework's source generator picks these up at compile time and embeds them in the generated help strings.

Example transformation:

**Before:**
```csharp
/// <summary>Set the active work item by ID or title pattern.</summary>
public async Task<int> Set([Argument] string idOrPattern, string output = ..., CancellationToken ct = default)
```

**After:**
```csharp
/// <summary>Set the active work item by ID or title pattern.</summary>
/// <param name="idOrPattern">Work item ID (e.g. 1234) or title substring to match.</param>
/// <param name="output">-o, Output format: human, json, jsonc, minimal.</param>
/// <param name="ct">Ignored.</param>
public async Task<int> Set([Argument] string idOrPattern, string output = ..., CancellationToken ct = default)
```

#### 4. Usage Examples (`CommandExamples` — new file `src/Twig/CommandExamples.cs`)

A static dictionary mapping command names to example strings, placed in a **separate file** to avoid further bloating `Program.cs` (already 734 lines). After `app.Run(args)` returns (which prints ConsoleAppFramework's help for `--help` invocations), we append examples:

```csharp
app.Run(args);

// Append usage examples when --help was requested
if (args.Any(a => a is "-h" or "--help") && args.Length >= 2)
    CommandExamples.ShowIfPresent(args);
```

`ShowIfPresent(string[] args)` resolves the command name internally — when `args.Length >= 2`, it tries `$"{args[0]} {args[1]}"` first (compound commands like `nav up`, `seed new`), then falls back to `args[0]`, then does nothing if neither matches. When `args.Length == 1`, it checks only `args[0]`. This handles the positional-argument pitfall naturally (e.g., `twig set 1234 --help` → `"set 1234"` won't match, falls back to `"set"` which does). **Hyphenated commands** like `flow-start` are single tokens — `args[0]` is `"flow-start"`, so the fallback to `args[0]` matches directly. The compound lookup `$"{args[0]} {args[1]}"` only matches space-separated sub-commands (e.g., `"nav up"`, `"seed new"`), which is correct because CAF uses `[Command("nav up")]` with spaces for sub-commands and `[Command("flow-start")]` as a single hyphenated name.

**Feasibility note**: ConsoleAppFramework's generated `Run` method calls `TryShowHelpOrVersion()` which prints help and returns — it does **not** call `Environment.Exit()`. This has been verified by examining the generated source output (the method uses a simple `if (TryShowHelpOrVersion(...)) return;` pattern). Post-run code executes reliably.

Each command gets ≥2 examples in the format:
```
Examples:
  twig set 1234              Set context by work item ID
  twig set "login page"      Set context by title pattern match
```

#### 5. Mark `seed` (bare) as `[Hidden]`

The bare `seed` command (`Seed()` method) is a backward-compat alias for `seed new`. It should be marked `[Hidden]` like the other backward-compat aliases (`up`, `down`, etc.) to prevent confusion in ConsoleAppFramework's default listing. *(See Background §Missing Commands Audit for the `Seed()` vs. `SeedNew()` parameter differences that are preserved.)*

### Data Flow

**`twig help` flow:**
```
args = ["help"] → early-exit block match → GroupedHelp.Show() → exit 0
```

**`twig help extra` flow (edge case):**
```
args = ["help", "extra"] → early-exit block miss (Length != 1)
  → IsKnownCommand("help") = true ("help" in KnownCommands)
  → app.Run(args) → CAF has no registered "help" command, so it shows its
    default flat help listing. This is an acceptable degradation for a
    nonsensical input — the user sees available commands either way.
```

**`twig <invalid>` flow (new):**
```
args = ["foobar"] → early-exit block miss → smart-landing miss → IsKnownCommand() = false
  → GroupedHelp.ShowUnknown("foobar") → stderr: "Unknown command: 'foobar'"
  → stdout: grouped help → exit 1
```

**`twig set --help` flow (enhanced):**
```
args = ["set", "--help"] → early-exit block miss → IsKnownCommand() = true
  → app.Run(args) → ConsoleAppFramework prints enhanced help with param descriptions
  → CommandExamples.ShowIfPresent(args) → prints examples → exit 0
```

### Design Decisions

| Decision | Rationale |
|----------|-----------|
| Pre-route interception vs. ConsoleAppFramework customization | ConsoleAppFramework's source generator produces help as compile-time string constants with no customization hooks. Pre-routing in `Program.cs` is the only way to control unknown-command output. |
| Manually-maintained `HashSet<string>` for KnownCommands | Reflection on `TwigCommands` works in tests but not in AOT production. Parsing the help text string is fragile. A static set is AOT-safe and acts as documentation. The maintenance cost is mitigated by the completeness test (T-1523-4) which fails the build on drift. |
| XML doc comments for param descriptions vs. custom help builder | ConsoleAppFramework natively supports `<param>` tags via source generator. This is the zero-overhead, AOT-compatible approach. Already proven by `Update` method's existing `<param>` tags for named options. Positional `[Argument]` parameters derive their display name from the C# parameter name; `<param>` adds description text. |
| Post-append examples vs. custom help interceptor | ConsoleAppFramework's generated `Run` uses `if (TryShowHelpOrVersion(...)) return;` — no `Environment.Exit()` — so post-run code executes reliably. Simple and maintainable. **Fragile dependency**: this relies on internal CAF behavior that is not part of the public API contract. The ConsoleAppFramework version is pinned in `Directory.Packages.props` and this behavior must be re-verified on upgrades (see Risks table). |
| `CommandExamples` in separate file vs. in `Program.cs` | `Program.cs` is already 734 lines. A separate `CommandExamples.cs` file keeps it focused on routing and reduces merge conflicts. |
| `seed` (bare) → `[Hidden]` | Aligns with the pattern used for other backward-compat aliases (`up`, `down`, etc.). `seed new` is the canonical command. *(See Background §Missing Commands Audit for preserved parameter differences.)* |
| Remove `save` from GroupedHelp; retain `sync` under Work Items | `Save()` is marked `[Hidden]` but was listed in help text. This inconsistency violates G-2 and confuses users about deprecated commands. The `sync` entry that follows in the Work Items category is retained — it serves a discoverability role for users browsing the Work Items section (distinct from the `sync` entry in "Getting Started" which serves onboarding). |
| Standardized `<param>` template for `output` | The `output` parameter appears on ~53 of 58 non-hidden methods with identical signature. Use a canonical description: `<param name="output">-o, Output format: human, json, jsonc, minimal.</param>`. The `ct` (CancellationToken) parameter is internal plumbing with no user-visible help text — omit it. **Note**: `GenerateDocumentationFile` is not enabled, so CS1573 does not fire under `TreatWarningsAsErrors`. |
| `query` under "Context" category | `query` is about finding work items in the current workspace, aligning with the "Context" category (alongside `set`, `show`, `web`). A new "Search" category would be a single-entry category, adding visual noise without discoverability benefit. |
| `"help"` included in `KnownCommands` | `twig help extra` (multi-arg) bypasses the `args.Length == 1` early-exit check. Without `"help"` in `KnownCommands`, it would trigger `ShowUnknown("help")` — a confusing error. Including it lets the command fall through to `app.Run()`. |

## Alternatives Considered

### 1. Reflection-based KnownCommands (rejected)

**Approach**: Use reflection on `TwigCommands` at startup to automatically build the command set.
**Pros**: Zero maintenance — always in sync with registered commands.
**Cons**: Incompatible with AOT (`PublishAot=true`). Reflection on source-generated types is unreliable under `TrimMode=full`. Would require `[DynamicallyAccessedMembers]` annotations that defeat the purpose of trimming.
**Decision**: Use a manually-maintained set with test-time validation.

### 2. ConsoleAppFramework unknown-command hook (rejected)

**Approach**: Hook into ConsoleAppFramework's built-in unknown command handling to inject grouped help.
**Pros**: Would avoid pre-routing logic entirely.
**Cons**: ConsoleAppFramework v5 has no extension point for customizing unknown command behavior. The flat help is baked into generated code with no override mechanism.
**Decision**: Pre-route interception before `app.Run(args)`.

### 3. Custom help rendering engine (rejected)

**Approach**: Replace ConsoleAppFramework's per-command help with a fully custom renderer.
**Pros**: Complete control over help format, including examples inline.
**Cons**: High maintenance burden. Duplicates work the source generator already does. Risks divergence from actual parameter definitions.
**Decision**: Use XML doc comments for param descriptions (source generator handles it) and append examples post-run.

- **ConsoleAppFramework v5.7.13**: XML `<param>` doc comments are supported for parameter descriptions and aliases. No version change needed.
- **No new NuGet packages**: All changes are within existing project boundaries.
- **No infrastructure changes**: Help system is purely local CLI rendering.
