# Global Process Configuration — Solution Design & Implementation Plan

> **Version**: 2.0 | **Date**: 2026-03-26 | **Owner**: dangreen
> **Tags**: feature, architecture, configuration, process-awareness

---

## Executive Summary

Twig's per-workspace `.twig/` configuration model requires users to repeat field selection ceremonies every time they initialize a new repository against the same Azure DevOps organization and process template. This design introduces a **three-layer configuration model** — workspace overrides > global profile > built-in defaults — centered on a `~/.twig/profiles/{org}/{process}/` global profile store. The global profile automatically captures user status-field selections via write-back from `twig config status-fields` and auto-applies them during `twig init` for matching org/process combinations. Hash-based staleness detection handles process template evolution gracefully through merge flows, while process-template-aware smart defaults and fill-rate-guided suggestions improve the first-run experience. The feature is purely additive — existing workspace-level behavior is unchanged, and global profiles act as a convenience cache that can be deleted without consequence.

## Background

### Current State

Twig is a .NET 10 Native AOT CLI tool (`PublishAot=true`, `JsonSerializerIsReflectionEnabledByDefault=false`) for managing Azure DevOps work items from the terminal. Its configuration lives in per-workspace `.twig/` directories:

- **`.twig/config`** — JSON file (`TwigConfiguration` POCO) with org, project, team, auth, display, git, and flow settings. Serialized via `TwigJsonContext` source generator.
- **`.twig/status-fields`** — Text-based field selection file controlling which extended ADO fields appear in `twig status`. Generated by `StatusFieldsConfig.Generate()`, edited by `twig config status-fields` through the user's `$EDITOR`.
- **`.twig/{org}/{project}/twig.db`** — SQLite cache with work items, field definitions, process types, navigation history, and pending changes.

The CLI follows a clean three-layer architecture:
- **Twig.Domain** — Pure domain logic (services, value objects, interfaces). No I/O dependencies.
- **Twig.Infrastructure** — ADO REST client, SQLite persistence, file-based config, Git CLI integration.
- **Twig** (CLI) — Commands, DI composition, formatters, hints, rendering.

Key architectural patterns:
- **Static sync services** (`FieldDefinitionSyncService`, `ProcessTypeSyncService`) with method parameters (not constructor-injected) for contexts where DI isn't available.
- **`TwigPaths`** as a centralized path-resolution utility with `SanitizePathSegment()` for filesystem safety.
- **`OutputFormatterFactory`** with human/JSON/minimal formatters — all user-facing output passes through formatters.
- **`HintEngine`** provides contextual suggestions suppressed for non-human output formats.
- **xUnit + Shouldly + NSubstitute** testing stack. Domain tests are pure unit tests; CLI tests use NSubstitute mocks for services; Infrastructure tests use temp directories.

### Motivation

Users working across multiple repositories within the same ADO organization experience friction: each `twig init` starts with a blank slate for field configuration, despite the field universe being identical for all projects using the same process template (Agile, Scrum, CMMI, Basic). The `twig config status-fields` ceremony must be repeated per-repo. This design eliminates that redundancy.

### Prior Art

- `StatusFieldsConfig.Generate(defs, existingContent)` already implements merge semantics — preserving user selections, appending new fields, dropping removed fields. This is the exact behavior needed for process evolution handling.
- `TwigPaths.SanitizePathSegment()` already handles filesystem-unsafe characters for the multi-context DB layout (`.twig/{org}/{project}/twig.db`).
- `DetectTemplateNameAsync()` in `IIterationService` heuristically identifies process templates by flagship work-item type names.

## Problem Statement

1. **Redundant configuration**: Users must run `twig config status-fields` and manually select fields in every new workspace, even when working with the same org and process template they've configured before.
2. **No preference persistence across workspaces**: Field selections, display preferences, and column configurations are entirely workspace-local. There is no mechanism to share them.
3. **Generic defaults**: The built-in `IsDefaultStarred()` heuristic uses generic keywords (effort, points, priority, severity, tags) and dateTime type matching. It doesn't leverage knowledge of specific ADO process templates to provide better initial selections.
4. **Silent field evolution**: When ADO process templates are customized and fields are added/removed, users have no signal that their status-fields configuration may be stale.

## Goals and Non-Goals

### Goals

| ID | Goal | Success Metric |
|----|------|---------------|
| G-1 | Global profile store at `~/.twig/profiles/` keyed by `(org, process-template)` | Profile directory created on first write-back |

## Requirements

### Functional Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| FR-01 | `~/.twig/profiles/{sanitized-org}/{sanitized-process}/` directory MUST be created lazily on first write-back | MUST |
| FR-02 | `GlobalProfileStore.LoadStatusFieldsAsync(org, process)` MUST return `null` when no profile exists (not throw) | MUST |
| FR-03 | `GlobalProfileStore.SaveStatusFieldsAsync(org, process, content)` MUST atomically write `status-fields` and `profile.json` | MUST |
| FR-04 | `FieldDefinitionHasher.ComputeFieldHash(definitions)` MUST produce a deterministic SHA-256 from sorted `(ReferenceName, DataType)` pairs | MUST |
| FR-05 | `InitCommand` MUST check for a global profile after detecting the process template | MUST |
| FR-06 | When global profile exists and hashes match, `InitCommand` MUST copy status-fields without user interaction | MUST |
| FR-07 | When global profile exists but hashes differ, `InitCommand` MUST merge using `StatusFieldsConfig.Generate(newDefs, existingProfileContent)` and update both workspace and global profile | MUST |
| FR-08 | `ConfigStatusFieldsCommand` MUST write-back to global profile after successful save | MUST |
| FR-09 | Write-back MUST be silent on failure — workspace save always succeeds regardless | MUST |
| FR-10 | `RefreshCommand` SHOULD recompute hashes after sync and update global profile metadata | SHOULD |
| FR-11 | `StatusFieldsConfig.IsDefaultStarred()` SHOULD use process-template-specific star lists when template name is known | SHOULD |
| FR-12 | Post-refresh hints SHOULD suggest unstarred high-fill-rate fields | SHOULD |

### Non-Functional Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| NFR-01 | All new code MUST be Native AOT compatible — no reflection, use `SHA256.HashData()` static method | MUST |
| NFR-02 | Global profile format MUST use existing `status-fields` text format — no new format | MUST |
| NFR-03 | `profile.json` MUST be serializable via `System.Text.Json` source generators (extend `TwigJsonContext`) | MUST |
| NFR-04 | Profile paths MUST be sanitized via `TwigPaths.SanitizePathSegment()` to prevent path traversal | MUST |
| NFR-05 | Global profile directory MUST use user-default file permissions (inherits umask) | MUST |

### Design Constraints

| ID | Constraint | Rationale |
|----|-----------|-----------|
| CON-01 | Follow `TwigPaths.SanitizePathSegment() + Path.Combine()` pattern for all path construction | Consistency with existing multi-context path layout |
| CON-02 | Follow `StatusFieldsConfig.Generate(defs, existingContent)` merge pattern | Proven, tested merge semantics already exist |
| CON-03 | Global profile is a convenience cache, not a contract — workspace `.twig/status-fields` always wins | Users must be able to diverge per-repo |
| CON-04 | Write-back is always automatic — no extra confirmation prompts | Feature purpose is reducing friction |

## Proposed Design

### Architecture Overview

```
┌─────────────────────────────────────────┐
│  Layer 3: Per-Workspace (.twig/)        │  Always wins — user can diverge per-repo
│  .twig/status-fields                    │  Existing behavior, unchanged semantics
│  .twig/config (Display section)         │
├─────────────────────────────────────────┤
│  Layer 2: Global Profile (~/.twig/)     │  NEW — shared across all workspaces
│  ~/.twig/profiles/{org}/{process}/      │  Keyed by (org, process-template)
│    status-fields                        │  User's field selection preferences
│    display-config.json                  │  Column prefs, fill-rate thresholds
│    profile.json                         │  Metadata: created, synced, hashes
├─────────────────────────────────────────┤
│  Layer 1: Built-in Defaults             │  Existing heuristics + new smart defaults
│  StatusFieldsConfig.DefaultStarKeywords │  effort, points, priority, severity, tags
│  + Process-aware star lists             │  Per-template curated field sets
└─────────────────────────────────────────┘

Resolution order: workspace → global profile → built-in defaults
```

### Key Components

#### 1. `GlobalProfilePaths` (Infrastructure — static utility)

**Location**: `src/Twig.Infrastructure/Config/GlobalProfilePaths.cs`
**Responsibility**: Resolves `(org, process)` tuples to filesystem paths under `~/.twig/profiles/`.

```csharp
public static class GlobalProfilePaths
{
    public static string GetProfileDir(string org, string process);
    public static string GetStatusFieldsPath(string org, string process);
    public static string GetDisplayConfigPath(string org, string process);
    public static string GetMetadataPath(string org, string process);
}
```

**Design rationale**: Static utility matching the `TwigPaths` pattern. Uses `Environment.GetFolderPath(Environment.SpecialFolder.UserProfile)` for cross-platform `~` resolution. All path segments sanitized via `TwigPaths.SanitizePathSegment()`.

#### 2. `ProfileMetadata` (Domain — value object)

**Location**: `src/Twig.Domain/ValueObjects/ProfileMetadata.cs`
**Responsibility**: Immutable record representing versioning and staleness data for a global profile.

```csharp
public sealed record ProfileMetadata(
    string Organization,
    string ProcessTemplate,
    DateTimeOffset CreatedAt,
    DateTimeOffset LastSyncedAt,
    string FieldDefinitionHash,
    string ProcessTypeHash,
    int FieldCount,
    int TypeCount);
```

**AOT**: Registered in `TwigJsonContext` with `[JsonSerializable(typeof(ProfileMetadata))]`.

#### 3. `FieldDefinitionHasher` (Domain — static service)

**Location**: `src/Twig.Domain/Services/FieldDefinitionHasher.cs`
**Responsibility**: Computes deterministic SHA-256 hashes for staleness detection.

```csharp
public static class FieldDefinitionHasher
{
    public static string ComputeFieldHash(IReadOnlyList<FieldDefinition> definitions);
    public static string ComputeProcessTypeHash(IReadOnlyList<ProcessTypeRecord> records);
}
```

**Algorithm**:
- `ComputeFieldHash`: Sort by `ReferenceName` (ordinal), concatenate `"{ReferenceName}:{DataType}\n"`, SHA-256 the UTF-8 bytes, return hex string prefixed with `sha256:`.
- `ComputeProcessTypeHash`: Sort by `TypeName` (ordinal), for each type sort state names ordinally, concatenate `"{TypeName}:{State1},{State2},...\n"`, SHA-256, return hex.

**AOT safety**: Uses `SHA256.HashData(byte[])` static method — no instance allocation, no trimming issues.

#### 4. `IGlobalProfileStore` (Domain — interface) + `GlobalProfileStore` (Infrastructure — implementation)

**Location**: `src/Twig.Domain/Interfaces/IGlobalProfileStore.cs`, `src/Twig.Infrastructure/Config/GlobalProfileStore.cs`

```csharp
public interface IGlobalProfileStore
{
    Task<string?> LoadStatusFieldsAsync(string org, string process, CancellationToken ct = default);
    Task SaveStatusFieldsAsync(string org, string process, string content, CancellationToken ct = default);
    Task<ProfileMetadata?> LoadMetadataAsync(string org, string process, CancellationToken ct = default);
    Task SaveMetadataAsync(string org, string process, ProfileMetadata metadata, CancellationToken ct = default);
    Task<DisplayConfig?> LoadDisplayConfigAsync(string org, string process, CancellationToken ct = default);
    Task SaveDisplayConfigAsync(string org, string process, DisplayConfig config, CancellationToken ct = default);
}
```

**Implementation strategy**:
- All `Load*` methods return `null` on missing file, corrupt JSON, or any I/O exception (FR-02, FR-09).
- All `Save*` methods: create directory tree lazily → write to `.tmp` file → `File.Move(overwrite: true)` for atomicity.
- Wrap all I/O in try-catch that swallows exceptions (FR-09). Callers don't need to handle failures.

#### 5. Process-template-aware smart defaults (Domain — `StatusFieldsConfig` modification)

**Location**: `src/Twig.Domain/Services/StatusFieldsConfig.cs`
**Change**: Add `GetProcessAwareDefaults(string? processTemplate)` and overload `IsDefaultStarred()`.

Curated reference-name sets per template:
- **Agile**: `Microsoft.VSTS.Common.Priority`, `Microsoft.VSTS.Scheduling.StoryPoints`, `Microsoft.VSTS.Common.ValueArea`, `System.Tags`, + dateTime fields
- **Scrum**: `Microsoft.VSTS.Scheduling.Effort`, `Microsoft.VSTS.Common.BusinessValue`, `Microsoft.VSTS.Common.BacklogPriority`, `System.Tags`, + dateTime fields
- **CMMI**: `Microsoft.VSTS.Common.Priority`, `Microsoft.VSTS.Scheduling.Size`, `Microsoft.VSTS.CMMI.Blocked`, `System.Tags`, + dateTime fields
- **Basic/unknown**: Existing keyword heuristic (unchanged)

### Data Flow

#### Init with Existing Profile (hashes match)

```
twig init --org myorg --project MyProject
  │
  ├─1. Create .twig/ + context dir (unchanged)
  ├─2. Detect process template → "Agile"
  ├─3. Fetch field definitions via FieldDefinitionSyncService.SyncAsync()
  ├─4. GlobalProfileStore.LoadMetadataAsync("myorg", "Agile")
  │      → ProfileMetadata { FieldDefinitionHash: "sha256:abc..." }
  ├─5. FieldDefinitionHasher.ComputeFieldHash(currentDefs)
  │      → "sha256:abc..." (matches)
  ├─6. GlobalProfileStore.LoadStatusFieldsAsync("myorg", "Agile")
  │      → "* Priority (Microsoft.VSTS.Common.Priority)  [integer]\n..."
  ├─7. File.WriteAllTextAsync(".twig/status-fields", profileContent)
  ├─8. "✓ Applied existing field configuration for myorg/Agile"
  └─9. Continue normal init (sprint fetch, etc.)
```

#### Init with Existing Profile (hashes differ — field evolution)

```
twig init --org myorg --project NewProject
  │
  ├─1–4. Same as above
  ├─5. FieldDefinitionHasher.ComputeFieldHash(currentDefs)
  │      → "sha256:def..." (differs from stored "sha256:abc...")
  ├─6. GlobalProfileStore.LoadStatusFieldsAsync("myorg", "Agile")
  │      → existing profile content
  ├─7. StatusFieldsConfig.Generate(currentDefs, profileContent)
  │      → merged content (preserves selections, adds new fields, drops removed)
  ├─8. File.WriteAllTextAsync(".twig/status-fields", mergedContent)
  ├─9. GlobalProfileStore.SaveStatusFieldsAsync("myorg", "Agile", mergedContent)
  ├─10. GlobalProfileStore.SaveMetadataAsync("myorg", "Agile", updatedMetadata)
  ├─11. "⚠ Process fields changed — merged with existing preferences"
  │     "  Run 'twig config status-fields' to review"
  └─12. Continue normal init
```

#### Config Status-Fields Write-Back

```
twig config status-fields
  │
  ├─1. Load field definitions from IFieldDefinitionStore (unchanged)
  ├─2. Generate/merge content, open in $EDITOR (unchanged)
  ├─3. Save to .twig/status-fields (unchanged)
  ├─4. ── NEW: Write-back ──
  │    ├─ Read org + process from TwigConfiguration
  │    ├─ Compute hashes (FieldDefinitionHasher)
  │    ├─ GlobalProfileStore.SaveStatusFieldsAsync(org, process, edited)
  │    ├─ GlobalProfileStore.SaveMetadataAsync(org, process, metadata)
  │    └─ "✓ Saved field preferences globally for myorg/Agile"
  └─5. Return 0
```

### Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| DD-01 | Key global profiles by `(org, process-template)` not `(org, project)` | Same process in same org shares the same field universe. Avoids N×M profile proliferation. Verified: `FieldDefinition` records from `GET /{project}/_apis/wit/fields` are identical across projects using the same process template. |
| DD-02 | Use `~/.twig/` not `%APPDATA%/twig/` or `~/.config/twig/` | Cross-platform consistency matching git convention (`~/.gitconfig`). `Environment.GetFolderPath(SpecialFolder.UserProfile)` resolves correctly on Windows (`%USERPROFILE%`), macOS, and Linux. |
| DD-03 | Automatic write-back with no confirmation prompt | Global profile is a convenience cache, not a contract. Extra prompts add friction to a feature whose purpose is reducing friction. |
| DD-04 | Hash-based staleness detection, not timestamp-based | Avoids false positives when process hasn't changed. Detects actual field set mutations. SHA-256 via `SHA256.HashData()` is AOT-safe. |
| DD-05 | Workspace layer always wins over global profile | Users MUST be able to diverge per-repo without affecting other workspaces. Global is the default seed, not a mandate. |
| DD-06 | Same `status-fields` text format in global profile | Reuses `StatusFieldsConfig.Generate()/Parse()` unchanged. No new format, no new parser. |
| DD-07 | Lazy directory creation on first write-back | `~/.twig/` should not exist on machines where the user hasn't configured field preferences yet. |
| DD-08 | `InitCommand` needs field definitions before profile check | Currently, `FieldDefinitionSyncService.SyncAsync()` runs during `RefreshCommand` (after init). For profile hash comparison, we need field defs during init. Solution: fetch field defs during init (after process template detection) and persist to DB, so they're available for hash computation. This also improves the first-use experience — `twig config status-fields` works immediately after `twig init` without needing `twig refresh` first. |
| DD-09 | `ConfigStatusFieldsCommand` needs `TwigConfiguration` and `IProcessTypeStore` injected | Currently uses primary constructor with 4 parameters. Add `IGlobalProfileStore`, `TwigConfiguration`, and `IProcessTypeStore` to support write-back. The command already has `IFieldDefinitionStore` for hash computation. |
| DD-10 | Register `IGlobalProfileStore` in `TwigServiceRegistration` (Infrastructure), not `CommandServiceModule` (CLI) | `GlobalProfileStore` is a pure file-system service with no CLI dependencies. Both CLI and TUI entry points should have access. Matches the pattern of `ISeedPublishRulesProvider` registration. |

## Alternatives Considered

| Alternative | Pros | Cons | Decision |
|-------------|------|------|----------|
| Key by `(org, project)` instead of `(org, process)` | Simpler mental model | Duplicates identical config for every project on same process; doesn't scale for large orgs | Rejected (DD-01) |
| Store in `%APPDATA%` / `~/.config/` | Follows XDG/Windows conventions | Platform-specific branching; git uses `~/.gitconfig` universally | Rejected (DD-02) |
| Explicit `twig config save-global` command | Clear user intent | Adds friction; users forget; defeats purpose of seamless reuse | Rejected (DD-03) |
| Timestamp-based staleness | Simple implementation | False positives when nothing changed; misses rapid changes | Rejected (DD-04) |
| Embedded profile in `.twig/config` JSON | Single file to manage | Not shareable across workspaces; defeats the purpose | Rejected |
| ADO Process API for real template ID | Accurate template identification | Extra API call; may require collection-admin permissions; blocked for some orgs | Deferred (OQ-01) |
| Fetch field defs lazily in init (only when profile exists) | Fewer API calls in first-init case | Inconsistent behavior; field defs useful anyway for immediate `twig config status-fields` | Rejected (DD-08) — always fetch |

## Impact Analysis

### Components Affected

| Component | Impact |
|-----------|--------|
| `InitCommand` | Moderate — new branch after template detection for profile check/apply/merge |
| `ConfigStatusFieldsCommand` | Moderate — new write-back logic after editor save |
| `RefreshCommand` | Low — hash recomputation and metadata update |
| `StatusFieldsConfig` | Low — new overload for process-aware defaults |
| `TwigJsonContext` | Trivial — add `[JsonSerializable]` attribute |
| `TwigServiceRegistration` | Trivial — add `IGlobalProfileStore` registration |

### Backward Compatibility

- **Fully backward compatible**. Absence of `~/.twig/profiles/` directory = current behavior.
- No changes to `.twig/status-fields` format, `.twig/config` schema, or SQLite DB schema.
- No changes to JSON/minimal output formats.
- Rollback: `rm -rf ~/.twig/profiles/` returns to pre-feature behavior.

### Performance

- **Init**: +1 file-system check (profile directory existence) + potential file read. Negligible.
- **Config status-fields**: +2 file writes (status-fields + profile.json) to `~/.twig/`. Sub-millisecond.
- **Refresh**: +SHA-256 hash computation over field definitions. ~50μs for typical 50-field set.

---

## Security Considerations

- **Data stored**: Field display names, reference names, data types, star/unstar preferences. No credentials, tokens, or PII. Organization name is already public (part of ADO URL).
- **Path traversal prevention**: All path segments sanitized via `TwigPaths.SanitizePathSegment()` which replaces `/ \ : * ? " < > |` with underscores and trims dots/whitespace.
- **File permissions**: `~/.twig/profiles/` inherits user-default permissions. No `chmod` needed — user-local, single-user access pattern.
- **No secrets**: Authentication tokens remain in the existing credential chain (Azure CLI, PAT env var).

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Process template detection heuristic maps two distinct custom templates to the same name (e.g., both → "Agile"), causing profile collision | Medium | Low | Hash-based staleness triggers merge flow on any field divergence, preserving user selections regardless of template name accuracy |
| File system permissions on `~/.twig/` vary (containers, shared machines, restricted environments) | Low | Low | FR-09: all global profile I/O is try-catch wrapped; workspace operations succeed regardless |
| Concurrent `twig init` / `twig config status-fields` from multiple terminals race on global writes | Low | Low | Last-writer-wins is acceptable for a user-local convenience cache; atomic write via `File.Move(overwrite: true)` prevents corruption |
| `TwigConfiguration` doesn't currently store detected process template name after init | N/A | Medium | DD-08: Store template name in `TwigConfiguration` during init so `ConfigStatusFieldsCommand` can read it for write-back |

## Implementation Phases

### Phase 1: Foundation (EPIC-001)
**Goal**: Build infrastructure for global profile storage, path resolution, and hash computation.
**Exit Criteria**: `GlobalProfilePaths`, `GlobalProfileStore`, `FieldDefinitionHasher`, and `ProfileMetadata` all implemented and tested. DI registration complete.

### Phase 2: Init Integration (EPIC-002)
**Goal**: `twig init` checks for and applies global profiles.
**Exit Criteria**: All three init branches (match, mismatch/merge, missing) work correctly. Field definitions fetched during init for hash computation.

### Phase 3: Write-Back (EPIC-003)
**Goal**: `twig config status-fields` auto-writes back to global profile.
**Exit Criteria**: Profile updated after every successful editor save. Failure is silent.

### Phase 4: Refresh Integration (EPIC-004)
**Goal**: `twig refresh` recomputes hashes and warns about field evolution.
**Exit Criteria**: Profile metadata updated after sync. Drift warning emitted when hashes change.

### Phase 5: Smart Defaults (EPIC-005)
**Goal**: Process-template-aware star lists and fill-rate suggestions.
