# Global Process Configuration — Solution Design & Implementation Plan

**Author:** dangreen + Copilot
**Status:** Complete
**Revision:** 3 — Codebase-grounded rewrite with exact file/line references

---

## Executive Summary

Twig's per-workspace `.twig/` configuration model requires users to repeat field selection ceremonies every time they initialize a new repository against the same Azure DevOps organization and process template. This design introduces a **three-layer configuration model** — workspace overrides > global profile > built-in defaults — centered on a `~/.twig/profiles/{org}/{process}/` global profile store. The global profile automatically captures user status-field selections via write-back from `twig config status-fields` and auto-applies them during `twig init` for matching org/process combinations. Hash-based staleness detection handles process template evolution gracefully through merge flows, while process-template-aware smart defaults and fill-rate-guided suggestions improve the first-run experience. The feature is purely additive — existing workspace-level behavior is unchanged, and global profiles act as a convenience cache that can be deleted without consequence.

## Background

### Current Architecture

Twig is a .NET 10 Native AOT CLI tool (`PublishAot=true`, `JsonSerializerIsReflectionEnabledByDefault=false`) for managing Azure DevOps work items from the terminal. Its configuration lives in per-workspace `.twig/` directories:

- **`.twig/config`** — JSON file (`TwigConfiguration` POCO, `TwigConfiguration.cs:12`) with Organization (L13), Project (L14), Team (L15), Auth, Defaults, Seed, Display, User, Git, Flow properties. Serialized via `TwigJsonContext` source generator (`TwigJsonContext.cs:17`).
- **`.twig/status-fields`** — Text-based field selection file. Generated by `StatusFieldsConfig.Generate()` (`StatusFieldsConfig.cs:75–90`), parsed by `StatusFieldsConfig.Parse()` (`StatusFieldsConfig.cs:95–120`), edited by `twig config status-fields` through the user's `$EDITOR` (`ConfigStatusFieldsCommand.cs:34–36`).
- **`.twig/{org}/{project}/twig.db`** — SQLite cache. Path resolved via `TwigPaths.GetContextDbPath()` (`TwigPaths.cs:56–59`).

The CLI follows a clean three-layer architecture:
- **Twig.Domain** — Pure domain logic. `FieldDefinition` is a sealed record at `FieldDefinition.cs:7–11` with `(ReferenceName, DisplayName, DataType, IsReadOnly)`.
- **Twig.Infrastructure** — ADO REST client, SQLite persistence, file-based config. `TwigServiceRegistration.AddTwigCoreServices()` (`TwigServiceRegistration.cs:39–108`) bootstraps DI.
- **Twig** (CLI) — Commands, formatters, hints, rendering.

Key integration points for this feature:
- **`TwigPaths.SanitizePathSegment()`** (`TwigPaths.cs:39–52`) — Replaces `/ \ : * ? " < > |` with underscores, trims whitespace and dots. Used for filesystem-safe org/project paths.
- **`StatusFieldsConfig.GenerateMerged()`** (`StatusFieldsConfig.cs:126–158`) — Preserves existing entry order and star/unstar selections, drops removed fields, appends new fields unmarked. This is the exact merge semantic needed for profile evolution.
- **`StatusFieldsConfig.IsDefaultStarred()`** (`StatusFieldsConfig.cs:57–68`) — Matches `DefaultStarKeywords` (`StatusFieldsConfig.cs:25–26`: `["effort", "points", "priority", "severity", "tags"]`) or `dateTime` data type.
- **`IIterationService.DetectTemplateNameAsync()`** (`IIterationService.cs:12`) — Heuristic process template detection by flagship work-item type names.
- **`FieldDefinitionSyncService.SyncAsync()`** (`FieldDefinitionSyncService.cs:7–20`) — Simple fetch-and-store; calls `iterationService.GetFieldDefinitionsAsync()` then `fieldDefinitionStore.SaveBatchAsync()`.

### Current Init Flow

`InitCommand.ExecuteAsync()` (`InitCommand.cs:58–205`):
1. **L63–66**: Derive context paths via `TwigPaths.ForContext(twigDir, org, project)`
2. **L117**: Build initial `TwigConfiguration` with org, project, team
3. **L131**: Write config early so iteration service can use it
4. **L137**: Detect process template via `iterationService.DetectTemplateNameAsync()`
5. **L140–156**: Fetch type appearances and area paths
6. **L211**: Save config, initialize SQLite, sync process types
7. No field definitions are fetched during init — that happens in `RefreshCommand` later
8. No `status-fields` file is created during init — user must run `twig config status-fields` manually

### Current Config Status-Fields Flow

`ConfigStatusFieldsCommand` (`ConfigStatusFieldsCommand.cs:13–17`) is a primary constructor with 4 dependencies: `IFieldDefinitionStore`, `IEditorLauncher`, `TwigPaths`, `OutputFormatterFactory`.

`ExecuteAsync()` (`ConfigStatusFieldsCommand.cs:19–43`):
1. **L24**: Fetch cached field definitions via `fieldDefinitionStore.GetAllAsync()`
2. **L30–32**: Read existing `.twig/status-fields` if present
3. **L34**: Call `StatusFieldsConfig.Generate(definitions, existingContent)`
4. **L36**: Launch editor via `editorLauncher.LaunchAsync(content)`
5. **L43**: Write result to `paths.StatusFieldsPath`
6. No write-back to any global profile — entirely workspace-local

### Current Refresh Flow

`RefreshCommand` (`RefreshCommand.cs:14–30`) injects 14 dependencies. At `RefreshCommand.cs:237–243`, field definitions are synced via `FieldDefinitionSyncService.SyncAsync()`. No hash computation or staleness detection occurs.

### Motivation

Users working across multiple repositories within the same ADO organization experience friction: each `twig init` starts with a blank slate for field configuration, despite the field universe being identical for all projects using the same process template (Agile, Scrum, CMMI, Basic). The `twig config status-fields` ceremony must be repeated per-repo.

### Prior Art

- `StatusFieldsConfig.Generate(defs, existingContent)` (`StatusFieldsConfig.cs:75–90`) already implements merge semantics. This is the exact behavior needed for process evolution handling.
- `TwigPaths.SanitizePathSegment()` (`TwigPaths.cs:39–52`) already handles filesystem-unsafe characters for the multi-context DB layout.
- `IIterationService.DetectTemplateNameAsync()` (`IIterationService.cs:12`) identifies process templates heuristically.

## Problem Statement

1. **Redundant configuration**: Users must run `twig config status-fields` and manually select fields in every new workspace, even when working with the same org and process template they've configured before.
2. **No preference persistence across workspaces**: Field selections, display preferences, and column configurations are entirely workspace-local. There is no mechanism to share them.
3. **Generic defaults**: `IsDefaultStarred()` (`StatusFieldsConfig.cs:57–68`) uses generic keywords. It doesn't leverage knowledge of specific ADO process templates to provide better initial selections.
4. **Silent field evolution**: When ADO process templates are customized and fields are added/removed, users have no signal that their status-fields configuration may be stale.

## Requirements

### Functional Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| FR-01 | `~/.twig/profiles/{sanitized-org}/{sanitized-process}/` directory MUST be created lazily on first write-back | MUST |
| FR-02 | `GlobalProfileStore.LoadStatusFieldsAsync(org, process)` MUST return `null` when no profile exists (not throw) | MUST |
| FR-03 | `GlobalProfileStore.SaveStatusFieldsAsync(org, process, content)` MUST atomically write `status-fields` and `profile.json` | MUST |
| FR-04 | `FieldDefinitionHasher.ComputeFieldHash(definitions)` MUST produce a deterministic SHA-256 from sorted `(ReferenceName, DataType)` pairs | MUST |
| FR-05 | `InitCommand` MUST check for a global profile after detecting the process template | MUST |
| FR-06 | When global profile exists and hashes match, `InitCommand` MUST copy status-fields without user interaction | MUST |
| FR-07 | When global profile exists but hashes differ, `InitCommand` MUST merge using `StatusFieldsConfig.Generate(newDefs, existingProfileContent)` and update both workspace and global profile | MUST |
| FR-08 | `ConfigStatusFieldsCommand` MUST write-back to global profile after successful save | MUST |
| FR-09 | Write-back MUST be silent on failure — workspace save always succeeds regardless | MUST |
| FR-10 | `RefreshCommand` SHOULD recompute hashes after sync and update global profile metadata | SHOULD |
| FR-11 | `StatusFieldsConfig.IsDefaultStarred()` SHOULD use process-template-specific star lists when template name is known | SHOULD |
| FR-12 | Post-refresh hints SHOULD suggest unstarred high-fill-rate fields | SHOULD |

### Non-Functional Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| NFR-01 | All new code MUST be Native AOT compatible — no reflection, use `SHA256.HashData()` static method | MUST |
| NFR-02 | Global profile format MUST use existing `status-fields` text format — no new format | MUST |
| NFR-03 | `profile.json` MUST be serializable via `System.Text.Json` source generators (extend `TwigJsonContext`) | MUST |
| NFR-04 | Profile paths MUST be sanitized via `TwigPaths.SanitizePathSegment()` to prevent path traversal | MUST |
| NFR-05 | Global profile directory MUST use user-default file permissions (inherits umask) | MUST |

### Design Constraints

| ID | Constraint | Rationale |
|----|-----------|-----------|
| CON-01 | Follow `TwigPaths.SanitizePathSegment() + Path.Combine()` pattern for all path construction | Consistency with existing multi-context path layout (`TwigPaths.cs:56–62`) |
| CON-02 | Follow `StatusFieldsConfig.Generate(defs, existingContent)` merge pattern | Proven, tested merge semantics at `StatusFieldsConfig.cs:75–90` |
| CON-03 | Global profile is a convenience cache, not a contract — workspace `.twig/status-fields` always wins | Users must be able to diverge per-repo |
| CON-04 | Write-back is always automatic — no extra confirmation prompts | Feature purpose is reducing friction |

## Proposed Design

### Architecture Overview

```
┌─────────────────────────────────────────┐
│  Layer 3: Per-Workspace (.twig/)        │  Always wins — user can diverge per-repo
│  .twig/status-fields                    │  Existing behavior, unchanged semantics
│  .twig/config (Display section)         │
├─────────────────────────────────────────┤
│  Layer 2: Global Profile (~/.twig/)     │  NEW — shared across all workspaces
│  ~/.twig/profiles/{org}/{process}/      │  Keyed by (org, process-template)
│    status-fields                        │  User's field selection preferences
│    display-config.json                  │  Column prefs, fill-rate thresholds
│    profile.json                         │  Metadata: created, synced, hashes
├─────────────────────────────────────────┤
│  Layer 1: Built-in Defaults             │  Existing heuristics + new smart defaults
│  StatusFieldsConfig.DefaultStarKeywords │  effort, points, priority, severity, tags
│  + Process-aware star lists             │  Per-template curated field sets
└─────────────────────────────────────────┘

Resolution order: workspace → global profile → built-in defaults
```

### Key Components

#### 1. `GlobalProfilePaths` (Infrastructure — static utility)

**Location**: `src/Twig.Infrastructure/Config/GlobalProfilePaths.cs`
**Pattern**: Mirrors `TwigPaths` (`TwigPaths.cs`) — static utility with `Path.Combine()` + `SanitizePathSegment()`.

```csharp
public static class GlobalProfilePaths
{
    private static readonly string HomeDir =
        Environment.GetFolderPath(Environment.SpecialFolder.UserProfile);

    public static string GetProfileDir(string org, string process)
        => Path.Combine(HomeDir, ".twig", "profiles",
            TwigPaths.SanitizePathSegment(org),
            TwigPaths.SanitizePathSegment(process));

    public static string GetStatusFieldsPath(string org, string process)
        => Path.Combine(GetProfileDir(org, process), "status-fields");

    public static string GetMetadataPath(string org, string process)
        => Path.Combine(GetProfileDir(org, process), "profile.json");
}
```

#### 2. `ProfileMetadata` (Domain — value object)

**Location**: `src/Twig.Domain/ValueObjects/ProfileMetadata.cs`
**Pattern**: Matches `FieldDefinition` (`FieldDefinition.cs:7–11`) — sealed record.

```csharp
public sealed record ProfileMetadata(
    string Organization,
    string ProcessTemplate,
    DateTimeOffset CreatedAt,
    DateTimeOffset LastSyncedAt,
    string FieldDefinitionHash,
    int FieldCount);
```

**AOT**: Registered in `TwigJsonContext` (`TwigJsonContext.cs`) with `[JsonSerializable(typeof(ProfileMetadata))]`.

#### 3. `FieldDefinitionHasher` (Domain — static service)

**Location**: `src/Twig.Domain/Services/FieldDefinitionHasher.cs`
**Pattern**: Matches `FieldDefinitionSyncService` (`FieldDefinitionSyncService.cs`) — static class with static methods.

```csharp
public static class FieldDefinitionHasher
{
    public static string ComputeFieldHash(IReadOnlyList<FieldDefinition> definitions)
    {
        // Sort by ReferenceName (ordinal), concatenate "ReferenceName:DataType\n",
        // SHA256.HashData() the UTF-8 bytes, return "sha256:{hex}"
    }
}
```

**Algorithm**: Sort by `ReferenceName` (ordinal), concatenate `"{ReferenceName}:{DataType}\n"`, `SHA256.HashData(Encoding.UTF8.GetBytes(combined))`, return hex prefixed with `sha256:`.

**AOT safety**: Uses `SHA256.HashData(byte[])` static method — no instance allocation, no trimming issues.

#### 4. `IGlobalProfileStore` + `GlobalProfileStore`

**Location**: `src/Twig.Domain/Interfaces/IGlobalProfileStore.cs`, `src/Twig.Infrastructure/Config/GlobalProfileStore.cs`
**Pattern**: Matches `ISeedPublishRulesProvider` / `FileSeedPublishRulesProvider` — domain interface in Twig.Domain, file-based implementation in Twig.Infrastructure.

```csharp
public interface IGlobalProfileStore
{
    Task<string?> LoadStatusFieldsAsync(string org, string process, CancellationToken ct = default);
    Task SaveStatusFieldsAsync(string org, string process, string content, CancellationToken ct = default);
    Task<ProfileMetadata?> LoadMetadataAsync(string org, string process, CancellationToken ct = default);
    Task SaveMetadataAsync(string org, string process, ProfileMetadata metadata, CancellationToken ct = default);
}
```

**Implementation strategy**:
- All `Load*` methods return `null` on missing file, corrupt JSON, or any I/O exception (FR-02, FR-09).
- All `Save*` methods: create directory tree lazily → write to `.tmp` file → `File.Move(overwrite: true)` for atomicity.
- Wrap all I/O in try-catch that swallows exceptions (FR-09). Callers don't need to handle failures.

#### 5. `TwigConfiguration.ProcessTemplate` property

**Location**: `src/Twig.Infrastructure/Config/TwigConfiguration.cs` — add after `Team` property (L15).

```csharp
public string ProcessTemplate { get; set; } = string.Empty;  // NEW — L16
```

**Backward-compatible**: Missing property in existing `.twig/config` JSON deserializes to `string.Empty` per `System.Text.Json` defaults. No schema version bump needed.

### Data Flow

#### Init with Existing Profile (hashes match)

```

  │
  ├─1. Create .twig/ + context dir (unchanged, InitCommand.cs:106–109)
  ├─2. Detect process template → "Agile" (InitCommand.cs:137)
  ├─3. NEW: Fetch field definitions via FieldDefinitionSyncService.SyncAsync()
  ├─4. NEW: GlobalProfileStore.LoadMetadataAsync("myorg", "Agile")
  │      → ProfileMetadata { FieldDefinitionHash: "sha256:abc..." }
  ├─5. NEW: FieldDefinitionHasher.ComputeFieldHash(currentDefs)
  │      → "sha256:abc..." (matches)
  ├─6. NEW: GlobalProfileStore.LoadStatusFieldsAsync("myorg", "Agile")
  │      → "* Priority (Microsoft.VSTS.Common.Priority)  [integer]\n..."
  ├─7. NEW: File.WriteAllTextAsync(contextPaths.StatusFieldsPath, profileContent)
  ├─8. "✓ Applied existing field configuration for myorg/Agile"
```

#### Init with Existing Profile (hashes differ — field evolution)

```
twig init --org myorg --project NewProject
  │
  ├─1–4. Same as above
  ├─5. FieldDefinitionHasher.ComputeFieldHash(currentDefs)
  │      → "sha256:def..." (differs from stored "sha256:abc...")
  ├─6. GlobalProfileStore.LoadStatusFieldsAsync("myorg", "Agile")
  │      → existing profile content
  ├─7. StatusFieldsConfig.Generate(currentDefs, profileContent) [StatusFieldsConfig.cs:75–90]
  │      → merged content (preserves selections, adds new fields, drops removed)
  ├─8. File.WriteAllTextAsync(".twig/status-fields", mergedContent)
  ├─9. GlobalProfileStore.SaveStatusFieldsAsync("myorg", "Agile", mergedContent)
  ├─10. GlobalProfileStore.SaveMetadataAsync("myorg", "Agile", updatedMetadata)
  ├─11. "⚠ Process fields changed — merged with existing preferences"
  │     "  Run 'twig config status-fields' to review"
  └─12. Continue normal init
```

```

  │
  ├─1. Load field definitions from IFieldDefinitionStore (unchanged, L24)
  ├─2. Generate/merge content, open in $EDITOR (unchanged, L30–36)
  ├─3. Save to .twig/status-fields (unchanged, L43)
  ├─4. ── NEW: Write-back ──

  │    ├─ GlobalProfileStore.SaveStatusFieldsAsync(org, process, edited)
  │    ├─ GlobalProfileStore.SaveMetadataAsync(org, process, metadata)
  │    └─ "✓ Saved field preferences globally for myorg/Agile"
  └─5. Return 0
```

### Design Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| DD-01 | Key global profiles by `(org, process-template)` not `(org, project)` | Same process in same org shares the same field universe. Avoids N×M profile proliferation. Verified: `FieldDefinition` records from `GET /{project}/_apis/wit/fields` are identical across projects using the same process template. |
| DD-02 | Use `~/.twig/` not `%APPDATA%/twig/` or `~/.config/twig/` | Cross-platform consistency matching git convention (`~/.gitconfig`). `Environment.GetFolderPath(SpecialFolder.UserProfile)` resolves correctly on Windows (`%USERPROFILE%`), macOS, and Linux. |
| DD-03 | Automatic write-back with no confirmation prompt | Global profile is a convenience cache, not a contract. Extra prompts add friction to a feature whose purpose is reducing friction. |
| DD-04 | Hash-based staleness detection, not timestamp-based | Avoids false positives when process hasn't changed. Detects actual field set mutations. SHA-256 via `SHA256.HashData()` is AOT-safe. |
| DD-05 | Workspace layer always wins over global profile | Users MUST be able to diverge per-repo without affecting other workspaces. Global is the default seed, not a mandate. |
| DD-06 | Same `status-fields` text format in global profile | Reuses `StatusFieldsConfig.Generate()/Parse()` (`StatusFieldsConfig.cs:75–120`) unchanged. No new format, no new parser. |
| DD-07 | Lazy directory creation on first write-back | `~/.twig/` should not exist on machines where the user hasn't configured field preferences yet. |
| DD-08 | Fetch field definitions during init, not just refresh | Currently `FieldDefinitionSyncService.SyncAsync()` only runs in `RefreshCommand` (`RefreshCommand.cs:237–243`). For profile hash comparison, we need field defs during init. Also improves first-use experience — `twig config status-fields` works immediately after `twig init`. |
| DD-09 | Add `ProcessTemplate` property to `TwigConfiguration` | `InitCommand` detects the template via `DetectTemplateNameAsync()` (`InitCommand.cs:137`) but only uses it transiently. `ConfigStatusFieldsCommand` and `RefreshCommand` need the template name for write-back. A new `ProcessTemplate` string property on `TwigConfiguration` (L16) is backward-compatible — missing key deserializes to `string.Empty`. |
| DD-10 | Expand `ConfigStatusFieldsCommand` primary constructor | Currently 4 dependencies (`ConfigStatusFieldsCommand.cs:13–17`). Add `IGlobalProfileStore` and `TwigConfiguration` for write-back. The command already has `IFieldDefinitionStore` for hash computation. |
| DD-11 | Register `IGlobalProfileStore` in `TwigServiceRegistration` | Matches the `ISeedPublishRulesProvider` / `FileSeedPublishRulesProvider` pattern (`TwigServiceRegistration.cs:95–99`). Both CLI and TUI entry points get access. |

## Alternatives Considered

| Alternative | Pros | Cons | Decision |
|-------------|------|------|----------|
| Key by `(org, project)` instead of `(org, process)` | Simpler mental model | Duplicates identical config for every project on same process; doesn't scale for large orgs | Rejected (DD-01) |
| Store in `%APPDATA%` / `~/.config/` | Follows XDG/Windows conventions | Platform-specific branching; git uses `~/.gitconfig` universally | Rejected (DD-02) |
| Explicit `twig config save-global` command | Clear user intent | Adds friction; users forget; defeats purpose of seamless reuse | Rejected (DD-03) |
| Timestamp-based staleness | Simple implementation | False positives when nothing changed; misses rapid changes | Rejected (DD-04) |
| Embedded profile in `.twig/config` JSON | Single file to manage | Not shareable across workspaces; defeats the purpose | Rejected |
| ADO Process API for real template ID | Accurate template identification | Extra API call; may require collection-admin permissions; blocked for some orgs | Deferred (OQ-01) |
| Fetch field defs lazily in init (only when profile exists) | Fewer API calls in first-init case | Inconsistent behavior; field defs useful anyway for immediate `twig config status-fields` | Rejected (DD-08) — always fetch |

- **Fully backward compatible**. Absence of `~/.twig/profiles/` directory = current behavior.
- New `ProcessTemplate` property in `.twig/config` deserializes to `string.Empty` when missing.

- **Init**: +1 ADO API call (`GetFieldDefinitionsAsync`) + 1 file-system check. Field defs are fast (~200ms).
- **Config status-fields**: +2 file writes to `~/.twig/`. Sub-millisecond.
- **Refresh**: +SHA-256 hash computation. ~50μs for typical 50-field set.

## Security Considerations

- **Data stored**: Field display names, reference names, data types, star/unstar preferences. No credentials, tokens, or PII.
- **Path traversal prevention**: All path segments sanitized via `TwigPaths.SanitizePathSegment()` (`TwigPaths.cs:39–52`) which replaces `/ \ : * ? " < > |` with underscores and trims dots/whitespace.
- **File permissions**: `~/.twig/profiles/` inherits user-default permissions. User-local, single-user access pattern.
- **No secrets**: Authentication tokens remain in the existing credential chain (Azure CLI, PAT env var).

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Process template detection heuristic maps two distinct custom templates to same name, causing profile collision | Medium | Low | Hash-based staleness triggers merge flow on any field divergence, preserving user selections regardless of template name accuracy |
